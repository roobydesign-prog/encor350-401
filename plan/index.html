Comprendre l'examen 350-401

Le 350-401 ENCOR (Implementing and Operating Cisco Enterprise Network Core Technologies) est l'examen « core » du CCNP Enterprise. Il valide ta maîtrise des technologies fondamentales des réseaux d'entreprise : architecture, virtualisation, infrastructure (switching + routage + services), assurance réseau, sécurité et automatisation. Il sert aussi d'examen de qualification pour le CCIE Enterprise Infrastructure.

Domaine 1 — Commutation d'entreprise (modules 1–7)

S1
MOD 01Architecture réseau d'entreprise Cisco

Cisco structure le réseau d'entreprise en blocs fonctionnels : Campus (accès, distribution, cœur), Edge (Internet/WAN), SP Edge et sites distants. Deux modèles de campus dominent :

  • 3-tier (accès / distribution / cœur) : pour les grands campus multi-bâtiments. Le cœur commute vite et ne fait rien d'autre (« switch fast, touch nothing »).
  • 2-tier / collapsed core : distribution et cœur fusionnés — le cas typique des PME et de la plupart des sites en Haïti.

Il faut aussi connaître les concepts modernes : Spine-Leaf pour les data centers, réseau en fabric (underlay physique + overlay logique), et le positionnement du cloud (on-prem vs cloud vs hybride).

L'accès = port utilisateur + PoE + sécurité de port ; la distribution = frontière L2/L3, politiques et agrégation ; le cœur = vitesse et redondance uniquement. C'est un classique de question drag-and-drop.
MOD 02Chemins de commutation Cisco (CEF, CAM, TCAM)

Trois mécanismes historiques de traitement des paquets :

  • Process switching : chaque paquet remonte au CPU. Lent — utilisé seulement pour le trafic destiné au routeur ou les cas exotiques.
  • Fast switching : le premier paquet est traité par le CPU, le résultat est mis en cache pour les suivants.
  • CEF (Cisco Express Forwarding) : le mode par défaut moderne. Deux tables construites à l'avance : la FIB (copie optimisée de la table de routage) et la table d'adjacence (réécritures L2 pré-calculées).

Côté matériel : la CAM stocke les correspondances exactes MAC→port (commutation L2) ; la TCAM permet des recherches avec masques (ACL, QoS, préfixes de routage) en un seul cycle.

Switch# show mac address-table dynamic
Router# show ip cef
Router# show adjacency detail
SVI vs port routé : un SVI (interface vlan 10) sert de passerelle pour tout un VLAN ; un port routé (no switchport) est une interface L3 point-à-point comme sur un routeur. L'examen adore tester la différence.
MOD 03Connectivité LAN de campus (VLAN, trunks, inter-VLAN)

Révision approfondie du CCNA avec le niveau de détail CCNP :

  • 802.1Q : tag de 4 octets inséré dans la trame ; le VLAN natif circule non taggé — il doit être identique des deux côtés du trunk.
  • DTP : négociation dynamique des trunks (dynamic auto, dynamic desirable). Bonne pratique : le désactiver (switchport nonegotiate) et configurer statiquement.
  • Routage inter-VLAN : router-on-a-stick (sous-interfaces) vs commutateur multicouche avec SVI — le second est la norme en entreprise.
Switch(config-if)# switchport mode trunk
Switch(config-if)# switchport trunk allowed vlan 10,20,99
Switch(config-if)# switchport trunk native vlan 99
Switch# show interfaces trunk
Mismatch de VLAN natif = les trames « fuient » d'un VLAN vers l'autre + messages CDP d'erreur. Symptôme classique dans les questions de dépannage.
MOD 04Topologies commutées redondantes (STP / RSTP / MST)

Sujet majeur de l'examen. Maîtrise l'élection et la convergence par cœur :

  • Élection du root bridge : priorité la plus basse (défaut 32768 + VLAN ID), puis MAC la plus basse. Manipulation : spanning-tree vlan 10 root primary.
  • Rôles de ports : root port (meilleur chemin vers le root), designated port (un par segment), blocked/alternate.
  • Coûts : basés sur la bande passante (méthode courte : 10 Gb=2, 1 Gb=4, 100 Mb=19).
  • RSTP (802.1w) : convergence en secondes grâce au mécanisme proposal/agreement ; états réduits à discarding/learning/forwarding ; rôles supplémentaires alternate et backup.
  • MST (802.1s) : regroupe plusieurs VLAN dans une même instance de spanning-tree — révision de la notion de région (nom + révision + table VLAN→instance identiques).
  • Mécanismes de protection : PortFast (accès direct au forwarding pour les ports terminaux), BPDU Guard (err-disable si BPDU reçu sur un port PortFast), Root Guard (empêche un switch en aval de devenir root), Loop Guard (protège contre les liens unidirectionnels).
Switch(config)# spanning-tree mode rapid-pvst
Switch(config)# spanning-tree vlan 1-99 root primary
Switch(config-if)# spanning-tree portfast
Switch(config-if)# spanning-tree bpduguard enable
Switch# show spanning-tree vlan 10
Sache lire un show spanning-tree et identifier : qui est root, pourquoi, quel port bloque et ce qui se passe si un lien tombe. C'est le scénario type des questions à sortie de commande.

MOD 06Agrégation de liens L2 (EtherChannel)

EtherChannel regroupe jusqu'à 8 liens physiques en un seul lien logique — STP le voit comme un seul port, donc pas de blocage entre les liens membres.

  • LACP (802.3ad, ouvert) : modes active (initie) et passive (répond). Au moins un côté doit être active.
  • PAgP (Cisco) : modes desirable / auto. Même logique.
  • Mode on : forcé sans négociation — les deux côtés doivent être on, sinon boucle possible.
  • Conditions : mêmes vitesse, duplex, VLAN autorisés, VLAN natif et mode sur tous les membres.
  • Load balancing : par hachage (src/dst MAC, IP ou port) — se vérifie avec show etherchannel load-balance.
Switch(config)# interface range g1/0/1-2
Switch(config-if-range)# channel-group 1 mode active
Switch# show etherchannel summary
active+passive = OK ; passive+passive = jamais de channel. desirable+auto = OK ; auto+auto = jamais. LACP et PAgP ne se parlent pas entre eux.
MOD 07Qualité de service (QoS)

La QoS gère la contention : quand le lien est saturé, qui passe en premier ?

  • Besoins applicatifs : la voix exige ≤150 ms de latence, ≤30 ms de gigue, ≤1 % de perte. La vidéo est gourmande et sensible, les données tolèrent.
  • Modèles : Best Effort (rien), IntServ (RSVP, réservation par flux — ne passe pas à l'échelle), DiffServ (marquage par classe — le standard réel).
  • Classification & marquage : CoS (L2, 3 bits 802.1p), DSCP (L3, 6 bits). Valeurs à connaître : EF (46) = voix, AF41 = vidéo interactive, CS0 = best effort.
  • Mécanismes de file : CBWFQ (bande passante garantie par classe), LLQ = CBWFQ + file prioritaire stricte pour la voix.
  • Policing vs shaping : le policing jette ou re-marque l'excès (entrée ou sortie) ; le shaping met en tampon et lisse (sortie seulement).
  • Évitement de congestion : WRED jette des paquets de façon sélective avant que la file ne déborde (évite la synchronisation globale TCP).
Policing = ciseaux (drop/remark), Shaping = entonnoir (buffer/délai). Le shaping ajoute de la latence, donc jamais pour la voix.

Domaine 2 — Routage (modules 8–13)

S2
MOD 08EIGRP expliqué (et comparé à OSPF)

À l'ENCOR, EIGRP est surtout testé en compréhension (la configuration avancée est pour l'ENARSI) :

  • Protocole à vecteur de distance avancé, algorithme DUAL, distance administrative 90 (interne) / 170 (externe).
  • Métrique composite : bande passante (la plus faible du chemin) + délai (cumulé) par défaut. K1=K3=1, les autres à 0.
  • Vocabulaire DUAL : Successor = meilleure route installée ; Feasible Successor = route de secours pré-validée (sa Reported Distance < Feasible Distance du successor — la condition de faisabilité). Convergence quasi instantanée si un FS existe.
  • vs OSPF : EIGRP converge très vite avec FS et supporte l'équilibrage à coût inégal (variance) ; OSPF est standard ouvert, vision complète de la topologie, meilleure interopérabilité multi-constructeur.
R1# show ip eigrp topology
R1# show ip eigrp neighbors
La condition de faisabilité (RD du voisin < FD actuelle) garantit l'absence de boucle sans recalcul. C'est LE concept EIGRP à connaître pour l'ENCOR.
MOD 09Implémentation d'OSPF

Le protocole le plus testé de l'examen. À maîtriser sur le bout des doigts :

Adjacences

  • États : Down → Init → 2-Way → ExStart → Exchange → Loading → Full.
  • Bloqué en 2-Way avec le DROTHER ? Normal sur un réseau multi-accès. Bloqué en ExStart/Exchange ? Problème de MTU. Jamais de voisin ? Vérifie : aire, timers hello/dead, masque de sous-réseau (sur broadcast), authentification, type de réseau.
  • DR/BDR : élection sur les segments multi-accès (priorité la plus haute, puis Router-ID le plus haut ; priorité 0 = jamais DR). Pas de préemption !

MOD 10Optimisation d'OSPF (+ OSPFv3)
  • Coût : bande passante de référence (100 Mb/s par défaut) ÷ bande passante du lien. Ajuste avec auto-cost reference-bandwidth 10000 (identique partout !) ou ip ospf cost.
  • Résumé de routes : inter-aires sur l'ABR (area 1 range 172.16.0.0 255.255.252.0) ; externes sur l'ASBR (summary-address).
  • Aires spéciales : stub (bloque les type 5, route par défaut injectée), totally stubby (bloque aussi les type 3), NSSA (stub qui autorise la redistribution locale via type 7).
  • Filtrage : distribute-list, filtrage de préfixes inter-aires sur l'ABR (area x filter-list).
  • Route par défaut : default-information originate (avec always pour l'annoncer même sans route par défaut locale).
  • OSPFv3 : pour IPv6 (et IPv4 en address-family). Config par interface (ipv6 ospf 10 area 0), Router-ID toujours au format 32 bits, voisinage via adresses link-local.
Le résumé OSPF ne se fait JAMAIS n'importe où : inter-aires = ABR seulement, externe = ASBR seulement. OSPF ne permet pas le résumé arbitraire comme EIGRP.
MOD 11eBGP

L'ENCOR se limite à l'essentiel d'eBGP (le reste est en ENARSI) :

  • Path vector, port TCP 179, voisins configurés manuellement (pas de découverte automatique). AD : eBGP = 20, iBGP = 200.
  • États : Idle → Connect → Active (⚠ Active = il essaie, pas qu'il marche !) → OpenSent → OpenConfirm → Established.
  • Attributs et ordre de sélection du meilleur chemin (mémorise le début) : Weight (Cisco, local, plus haut) → Local Preference (plus haut, dans l'AS) → routes locales → AS-Path le plus court → Origin (IGP<EGP<Incomplete) → MED le plus bas → eBGP sur iBGP → …
  • Annonce de réseaux : network exige que la route existe exactement dans la table de routage.
R1(config)# router bgp 65001
R1(config-router)# neighbor 203.0.113.2 remote-as 65002
R1(config-router)# network 198.51.100.0 mask 255.255.255.0
R1# show ip bgp summary
R1# show ip bgp
Moyen mnémotechnique pour le best path : « We Love Oranges AS Oranges Mean Pure Refreshment » (Weight, LocalPref, Originated, AS-Path, Origin, MED…).
MOD 12Protocoles multicast
  • Adressage : 224.0.0.0/4. Réservées : 224.0.0.1 (tous les hôtes), 224.0.0.2 (tous les routeurs), 224.0.0.5/6 (OSPF), 224.0.0.9 (RIPv2), 224.0.0.13 (PIM).
  • IGMP : entre l'hôte et le routeur — l'hôte « s'abonne » à un groupe (IGMPv2 : join/leave ; IGMPv3 : filtrage par source pour SSM).
  • IGMP snooping : le commutateur écoute IGMP pour n'envoyer le flux qu'aux ports abonnés.
  • PIM : entre routeurs. Dense mode (flood & prune — rare), Sparse mode (arbre partagé enraciné sur un RP — Rendezvous Point, puis bascule possible vers l'arbre source SPT), PIM-SSM (l'hôte précise la source, pas besoin de RP).
  • RPF check : un routeur n'accepte un flux multicast que s'il arrive par l'interface qui mène à la source (anti-boucle).
Sparse mode + RP = le modèle par défaut à connaître. SSM (232.0.0.0/8) supprime le besoin de RP car le récepteur connaît déjà la source.
MOD 13NAT et NVI
  • Terminologie : inside local (IP privée réelle), inside global (IP publique traduite), outside local/global. Les questions jouent sur ces 4 termes.
  • Types : NAT statique (1:1), NAT dynamique (pool), PAT / overload (n:1 avec ports — le plus courant).
  • NVI (NAT Virtual Interface) : supprime la notion inside/outside — on met ip nat enable sur les interfaces et la traduction fonctionne dans n'importe quel sens (utile en NAT entre VRF ou topologies complexes).

R1(config-if)# ip nat outside    ! WAN

MOD 14Sécurité de l'infrastructure (ACL, CoPP)
  • ACL numérotées : standard 1–99 (source seulement, à placer près de la destination), étendues 100–199 (source+dest+protocole+ports, à placer près de la source). Préfère les ACL nommées, éditables ligne par ligne.
  • Deny implicite à la fin de toute ACL — toujours.
  • ACL sur VTY : access-class 10 in pour restreindre SSH/Telnet.
  • CoPP (Control Plane Policing) : protège le CPU du routeur en limitant le débit du trafic destiné au plan de contrôle (SSH, SNMP, protocoles de routage, ICMP). Construction MQC : class-map → policy-map avec police → service-policy input sous control-plane.
R1(config)# ip access-list extended PROTECT-SSH
R1(config-ext-nacl)# permit tcp 10.0.0.0 0.0.0.255 any eq 22
R1(config)# class-map CM-SSH
R1(config-cmap)# match access-group name PROTECT-SSH
R1(config)# policy-map PM-COPP
R1(config-pmap-c)# police 64000 conform-action transmit exceed-action drop
R1(config)# control-plane
R1(config-cp)# service-policy input PM-COPP
CoPP s'applique en input sous control-plane, pas sur une interface physique. Et attention : trop restrictif, tu casses tes propres adjacences OSPF/BGP.
MOD 15Architecture de sécurité d'entreprise
  • Menaces : DoS/DDoS, spoofing, man-in-the-middle, phishing, malware, exploitation de vulnérabilités (comprendre le vocabulaire : menace / vulnérabilité / exploit / risque).
  • Composants Cisco à connaître (rôle, pas config) : pare-feu nouvelle génération (Firepower NGFW — inspection stateful + applicatif + IPS), Cisco Umbrella (sécurité DNS), Cisco Secure Endpoint (AMP), ESA/WSA (mail/web), Cisco ISE (politique d'accès, 802.1X, profilage), Stealthwatch (analyse NetFlow), TrustSec (segmentation par SGT), MACsec (chiffrement L2, 802.1AE).
  • Défense en profondeur : plusieurs couches — périmètre, réseau, endpoint, identité.
Question type : « quel produit répond à ce besoin ? ». Retiens la fonction d'une phrase pour chaque produit — ISE = qui accède ; Umbrella = filtrage DNS ; Stealthwatch = détection par télémétrie ; TrustSec = segmentation par tags SGT indépendante des VLAN/IP.

MOD 17Virtualisation : protocoles et techniques
  • Virtualisation des serveurs : hyperviseur de type 1 (bare-metal : ESXi, KVM, Hyper-V) vs type 2 (sur OS hôte : VirtualBox, VMware Workstation). VM vs conteneurs (les conteneurs partagent le noyau — plus légers).
  • vSwitch : commutateur virtuel reliant les VM aux NIC physiques ; SR-IOV pour l'accès direct au matériel.
  • VRF (Virtual Routing and Forwarding) : plusieurs tables de routage indépendantes sur le même routeur — la segmentation L3 de référence. VRF-lite = sans MPLS.
  • Virtualisation des fonctions réseau (NFV) : routeurs/pare-feu virtuels (Catalyst 8000V, etc.).

R1(config-vrf)# address-family ipv4

R1# show ip route vrf CLIENTS
R1# ping vrf CLIENTS 10.1.1.2
Appliquer vrf forwarding sur une interface efface son adresse IP — il faut la reconfigurer après. Détail vicieux que l'examen adore.
MOD 18VPN et interfaces virtuelles (GRE, IPsec, LISP, VXLAN)
  • GRE : tunnel simple multi-protocoles (supporte le multicast → les IGP passent dedans !), mais aucun chiffrement. Overhead : 24 octets. Attention au MTU (1476) et au routage récursif.
  • IPsec : chiffrement (ESP), intégrité, authentification. Deux phases IKE : phase 1 = tunnel de gestion (ISAKMP), phase 2 = tunnel de données. Modes transport vs tunnel. IPsec seul ne transporte pas le multicast → d'où le combo GRE over IPsec.
  • Types de VPN : site-à-site, accès distant (SSL/TLS ou IPsec), DMVPN (hub-and-spoke dynamique — concept), FlexVPN (concept).
  • LISP : sépare l'identité (EID) de la localisation (RLOC). Composants : ITR/ETR (routeurs d'entrée/sortie), Map Server / Map Resolver (l'« annuaire »). C'est le plan de contrôle de SD-Access.
  • VXLAN : encapsulation L2-dans-UDP (port 4789) avec un identifiant VNI sur 24 bits (≈ 16 millions de segments vs 4096 VLAN). Les extrémités = VTEP. C'est le plan de données de SD-Access.
R1(config)# interface Tunnel0
R1(config-if)# ip address 10.255.0.1 255.255.255.252
R1(config-if)# tunnel source g0/0/0
R1(config-if)# tunnel destination 203.0.113.10
Formule d'examen : SD-Access = LISP (contrôle) + VXLAN (données) + TrustSec/SGT (politique). Retiens ce trio, il revient partout.
MOD 19Services réseau (NTP, syslog, SNMP, NetFlow, DHCP…)
  • NTP : hiérarchie en stratum (0 = horloge de référence, 1 = serveur directement connecté…). UDP 123. Un routeur peut être client, serveur ou pair.
  • Syslog : niveaux 0–7 à connaître par cœur : 0 Emergency, 1 Alert, 2 Critical, 3 Error, 4 Warning, 5 Notification, 6 Informational, 7 Debugging. Mnémonique : « Every Awesome Cisco Engineer Will Need Ice-cream Daily ».
  • SNMP : v2c (community strings, en clair) vs v3 (authentification + chiffrement : modes noAuthNoPriv / authNoPriv / authPriv). Traps (non acquittés) vs informs (acquittés).
  • NetFlow / Flexible NetFlow : télémétrie des flux. FNF = 3 blocs : flow record (quoi collecter) + flow exporter (où envoyer) + flow monitor (appliqué à l'interface).
  • DHCP relay : ip helper-address transforme le broadcast DHCP en unicast vers le serveur.
  • IP SLA : sondes actives (icmp-echo, jitter…) pour mesurer et déclencher (tracking + FHRP ou routes statiques conditionnelles).
R1(config)# ntp server 216.239.35.0
R1(config)# logging host 10.0.0.100
R1(config)# logging trap warnings          ! envoie niveaux 0-4
R1(config)# ip sla 1
R1(config-ip-sla)# icmp-echo 8.8.8.8
R1(config)# ip sla schedule 1 life forever start-time now
logging trap warnings envoie les niveaux 0 à 4 inclus (le niveau choisi ET tout ce qui est plus grave). Beaucoup de candidats inversent.
MOD 20Outils d'analyse et de dépannage
  • Méthodologie : définir le problème → collecter → analyser → éliminer → proposer une hypothèse → tester → documenter.
  • ping / traceroute étendus : choisir la source, la taille, le DF-bit (diagnostic MTU !).
  • debug : puissant mais dangereux en production — utiliser des ACL de filtrage (debug ip packet 101) et undebug all.
  • SPAN / RSPAN / ERSPAN : copie du trafic vers un analyseur. SPAN = local ; RSPAN = via un VLAN dédié entre switches ; ERSPAN = encapsulé GRE à travers un réseau L3.
  • Conditional debugging : debug platform condition sur IOS-XE pour cibler une interface/adresse.

Switch(config)# monitor session 1 destination interface g1/0/24

MOD 21Bases de Python pour le réseau

Tu n'écriras pas de script à l'examen : tu dois lire et prédire ce que fait un extrait de code.

  • Types : str, int, list [ ], dict { } (clé:valeur — exactement la structure du JSON !), bool.
  • Structures : if/elif/else, boucles for, fonctions def. L'indentation EST la syntaxe.
  • Manipulation JSON : import json, json.loads() (texte → dict), json.dumps() (dict → texte).
  • Requêtes API : bibliothèque requests — requests.get(url, headers=...), code 200 = OK.
import json, requests
r = requests.get("https://sandbox/api/v1/devices",
                 headers={"X-Auth-Token": token}, verify=False)
devices = r.json()
for d in devices["response"]:
    print(d["hostname"], d["managementIpAddress"])
Entraîne-toi à lire du JSON imbriqué et à écrire le « chemin » d'une valeur : data["response"][0]["hostname"]. C'est le format de question le plus fréquent du domaine 6.0.
MOD 22Protocoles de programmabilité (YANG, NETCONF, RESTCONF, EEM…)
  • YANG : langage de modélisation des données de configuration/état (le « schéma »). Ce n'est pas un protocole de transport.
  • NETCONF : transport SSH port 830, encodage XML, opérations get-config / edit-config, notion de datastores (running, candidate, startup).
  • RESTCONF : transport HTTPS (443), verbes REST (GET/POST/PUT/PATCH/DELETE), encodage JSON ou XML, URI basées sur les modèles YANG.
  • APIs REST : northbound (intention → contrôleur, ex. API Catalyst Center) vs southbound (contrôleur → équipements : NETCONF, RESTCONF, SNMP, CLI). Codes HTTP : 200 OK, 201 Created, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 500 Server Error.
  • Authentification API : Basic (user:pass en Base64), Bearer token, clés API.
  • EEM (Embedded Event Manager) : automatisation embarquée sur IOS — un event (syslog, timer, CLI…) déclenche des actions.
  • Outils de config management (concepts) : Ansible (sans agent, SSH, playbooks YAML, push), Terraform (infrastructure as code déclarative, fichiers HCL, plan/apply), Puppet/Chef (avec agent, pull).
event manager applet SAUVEGARDE
 event syslog pattern "%SYS-5-CONFIG_I"
 action 1.0 cli command "enable"
 action 2.0 cli command "copy running-config tftp://10.0.0.100/backup.cfg"
Trio à ne jamais confondre : YANG = le modèle ; NETCONF = SSH 830 + XML ; RESTCONF = HTTPS + JSON/XML + verbes REST. Trois questions faciles si c'est clair, trois pièges sinon.

Domaine 6 — SDN : Catalyst Center, SD-Access, SD-WAN (modules 23–26) + sans-fil

S4
MOD 23Cisco Catalyst Center (ex-DNA Center)

Le contrôleur central des réseaux d'entreprise Cisco. Quatre grands workflows :

  • Design : hiérarchie de sites, profils réseau, pools IP, paramètres globaux.
  • Policy : politiques d'accès basées sur l'identité (groupes → SGT via ISE).
  • Provision : déploiement des configurations, Plug-and-Play, LAN automation.
  • Assurance : santé du réseau et des clients, télémétrie, analyse des problèmes avec IA, « time travel » pour rejouer un incident.

Catalyst Center expose une API northbound REST (Intent API) et pilote les équipements par les interfaces southbound (NETCONF, RESTCONF, SNMP, CLI).

Assurance = la fonctionnalité différenciante à l'examen : c'est elle qui répond aux questions du type « quel outil montre la santé des clients et l'historique d'un problème ? »
MOD 24Cisco SD-Access

SD-Access = le campus en mode fabric, piloté par Catalyst Center + ISE.

  • Underlay : réseau IP routé classique (souvent IS-IS via LAN automation) — il transporte, c'est tout.
  • Overlay (le fabric) : LISP = plan de contrôle (annuaire qui-est-où), VXLAN = plan de données (encapsulation qui transporte aussi le SGT), TrustSec/SGT = plan de politique.
  • Rôles des nœuds :
    • Control Plane Node : la base LISP (Map Server/Resolver) — qui répond « cet hôte est derrière tel edge ».
    • Edge Node : équivalent du switch d'accès — encapsule/désencapsule VXLAN.
    • Border Node : la porte vers l'extérieur du fabric (data center, Internet, WAN).
    • Fabric WLC / AP : le sans-fil intégré au fabric (le trafic des AP entre dans VXLAN à l'edge).
  • Bénéfices : segmentation par identité (macro : VN/VRF ; micro : SGT), mobilité, politique unique filaire + sans-fil.
Question drag-and-drop très fréquente : associer Control Plane Node / Edge / Border à leur définition. Et retiens : underlay = IS-IS, overlay contrôle = LISP, overlay données = VXLAN, politique = SGT.

Le WAN piloté par contrôleurs avec quatre plans distincts :

MOD 26APIs de Catalyst Center et vManage
  • Northbound : l'Intent API de Catalyst Center et l'API REST de vManage — on exprime l'intention (« crée ce SSID », « donne-moi la liste des équipements ») en JSON via HTTPS.
  • Southbound : les protocoles vers les équipements : NETCONF, RESTCONF, SNMP, SSH/CLI (et OMP côté SD-WAN).

BONUSSans-fil (transversal — Architecture + Infrastructure + Sécurité)

Le Wi-Fi n'a pas de module dédié dans le plan NetAcad mais pèse dans le blueprint. À couvrir en semaine 4 :

  • Notions RF : dBm, RSSI, SNR, canaux 2,4 GHz (1/6/11) vs 5 GHz vs 6 GHz, interférences ; types d'antennes (omnidirectionnelle vs directionnelle : Yagi, patch, parabole).
  • Modes de déploiement : autonome, centralisé (WLC + AP en mode local, tunnel CAPWAP), FlexConnect (commutation locale au site distant), fabric (SD-Access), cloud (Meraki, Catalyst 9800-CL).
  • CAPWAP : UDP 5246 (contrôle) / 5247 (données). Découverte du WLC : broadcast, DHCP option 43, DNS (CISCO-CAPWAP-CONTROLLER), mémoire.
  • Roaming : intra-contrôleur vs inter-contrôleurs (L2 vs L3), mobility groups.
  • Localisation : triangulation RSSI, rôle de Cisco Spaces/MSE.
  • Sécurité Wi-Fi : WPA2/WPA3, PSK vs 802.1X/EAP (EAP-TLS = certificats des deux côtés, PEAP = certificat serveur + mot de passe client), WebAuth invités.
AP en mode local : TOUT le trafic client remonte au WLC en CAPWAP. FlexConnect : le trafic peut sortir localement — crucial pour les sites distants avec WAN limité (scénario très haïtien !).

Plan de labs pratiques (CML-Free / Packet Tracer)

La topologie de référence du cours (R1-R2-R3 + D1-D2 + A1) se reproduit très bien en CML-Free. Priorise ces 10 labs — un par soirée environ :

  1. STP : forcer le root sur D1, observer les ports bloqués, couper un lien, chronométrer la convergence RSTP.
  2. EtherChannel : LACP entre D1-D2, provoquer volontairement un mismatch (mode, VLAN) et lire show etherchannel summary (flags !).
  3. HSRP : D1/D2 en passerelles, priorité + preempt + tracking d'interface, tester le basculement en ping continu.
  4. OSPF multi-aires : aire 0 + aire 1 stub, résumé sur l'ABR, lire la LSDB (show ip ospf database) et identifier chaque type de LSA.
  5. OSPFv3 : la même topologie en IPv6.
  6. eBGP : R1 (AS 65001) ↔ R2 (AS 65002), annonce de réseaux, lire show ip bgp (les codes > et *).
  7. GRE + OSPF dans le tunnel : R1–R3 via R2, puis protéger avec IPsec.
  8. VRF-lite : deux clients isolés sur le même routeur, ping par VRF.
  9. NAT/PAT + IP SLA : PAT vers l'extérieur + sonde icmp-echo avec route statique trackée.
  10. CoPP + AAA : policer SSH/ICMP vers le control-plane ; AAA avec fallback local.
Pour SD-Access, SD-WAN et Catalyst Center : pas de lab possible en CML-Free — utilise les sandboxes Cisco DevNet (gratuites, réservables) : « Catalyst Center Always-On » et « SD-WAN Always-On ». Fais-y au moins une session d'appels API avec Postman ou Python : tu retiendras 10× mieux le flux token → requêtes.
